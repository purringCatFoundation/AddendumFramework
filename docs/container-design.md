# Service Container Design

This document describes the intended container direction. It is not implemented yet.

## Goals

- Keep HTTP runtime PSR-first and free from reflection, file scanning and attribute reads.
- Use `Psr\Container\ContainerInterface` as the runtime lookup boundary when container support is introduced.
- Compile dependency metadata during cache warmup.
- Allow actions, middleware, guardians and validators to receive dependencies without hand-written factories for every class.
- Keep direct `new` fallback only for classes with no dependencies and only where it is explicitly part of the compiled metadata.

## Non-Goals

- Do not introduce a framework-specific service locator API.
- Do not make middleware options carry service instances.
- Do not inspect constructors in HTTP runtime.
- Do not require a container for the first compiled routing/cache iteration.

## Runtime Shape

HTTP bootstrap should eventually load a compiled application file and construct:

```php
$container = require __DIR__ . '/container.php';
$routes = (require __DIR__ . '/routes.php')();

return new App(
    new Router($routes),
    $container->get(LoggerInterface::class),
    $container->get(HttpCacheRuntime::class),
    new ActionRequestHandlerFactory($container)
);
```

The concrete container may be generated by Addendum or delegated to an adapter around an existing PSR-11 container. Runtime code should only depend on `Psr\Container\ContainerInterface`.

## Compiled Metadata

Warmup should generate dependency metadata alongside routes:

```php
return [
    'services' => [
        SomeAction::class => [
            'shared' => false,
            'constructor' => [SomeRepository::class, ClockInterface::class],
        ],
        SomeRepository::class => [
            'shared' => true,
            'constructor' => [Connection::class],
        ],
    ],
    'aliases' => [
        LoggerInterface::class => MonologLogger::class,
        ClockInterface::class => SystemClock::class,
    ],
];
```

Reflection is allowed while producing this file. Runtime must consume the compiled graph directly.

## Container Responsibilities

- Resolve aliases from interface to concrete class.
- Instantiate services from compiled constructor dependency lists.
- Cache shared services.
- Detect cycles during warmup where possible and at runtime as a safety net.
- Throw PSR-11 compatible `NotFoundExceptionInterface` and `ContainerExceptionInterface` errors.

## Integration Points

- `ActionRequestHandlerFactory` should ask the container for the action and middleware factory when a compiled dependency graph exists.
- `AccessControl` should ask the container for guardian instances only after container support lands.
- `ValidateRequestAttribute` should keep using `ValidatorDefinition` for validators; definitions can later reference container service IDs if validators gain dependencies.
- `HttpCacheRuntime`, logger and database connections should be normal container services.

## Phased Implementation

1. Generate `dependencies.php` with constructor metadata and aliases during `cache:warmup`.
2. Generate `container.php` that returns a PSR-11 container backed by the compiled metadata.
3. Change compiled `app.php` to load the container and pass it to factories that need it.
4. Update action, middleware and guardian creation to use the container.
5. Keep dynamic CLI paths independent from compiled container files.

## Open Questions

- Whether Addendum should ship its own small compiled container or only provide an adapter layer.
- How users should define aliases and scalar configuration values.
- Whether service sharing should default to shared or non-shared for actions and middleware.
- How much of the dependency graph should be included in `cache:debug --details`.
